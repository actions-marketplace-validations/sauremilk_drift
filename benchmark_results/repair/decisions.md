# Design Decisions — Repair Benchmark

## D1: Synthetic Repos Modeled After Real Findings

**Decision**: Create synthetic repos whose drift patterns mirror actual
findings from `flask_full.json` and `httpx_full.json`, rather than
running repairs against the real cloned repos.

**Reason**: Real repos require network access, are mutable (new commits
change results), and repairs would need to understand upstream project
intent.  Synthetic repos give deterministic control while being
grounded in real-world finding structures.  Phase B validates against
real data as a cross-check.

## D2: Verification via Specific Finding + Score, not just Count

**Decision**: A repair is verified by checking both whether the
*specific* finding disappears (`_find(post, signal, keyword) is None`)
*and* whether the overall signal count or drift score decreases.

**Reason**: Signal count alone is misleading — fixes can introduce new
findings (e.g., splitting a complex function into helpers may create new
EDS findings for the helpers).  The targeted keyword check proves the
*intended* issue was resolved; the score delta proves measurable
improvement.

## D3: Exactly one Incorrect Repair (Failure Case)

**Decision**: Include exactly one incorrect repair — renaming a
duplicate function without consolidating the body.

**Reason**: One failure case suffices to prove that verification is not
trivially satisfiable.  MDS detects structural duplication via body
hashes, not names.  A rename leaves the hash identical, so the finding
persists.  This directly demonstrates Drift's hash-based detection and
that agent-tasks do not merely suggest surface-level renames.

## D4: Task Quality Validation as a Separate Check

**Decision**: `_validate_quality()` checks schema completeness, priority
ordering, ID format (signal prefix + hash), and non-empty fields before
any repair is attempted.

**Reason**: Separating quality validation from causal verification
creates a clear two-stage gate:
1. Are the tasks *well-formed*? (quality)
2. Do the tasks actually *work*? (causality)

If quality fails, the repair cycle need not run.

## D5: Priority = severity_weight × max(impact, score)

**Decision**: Verify that tasks are sorted by descending priority using
the formula `severity_weight × max(impact_score, signal_score)`.

**Reason**: This formula is encoded in `agent_tasks.py` and determines
the order in which an agent should address issues.  A malformed priority
could cause an agent to fix cosmetic issues before critical ones.

## D6: Extended Prefix Set with Fallback

**Decision**: ID prefix validation uses a set that includes both
canonical prefixes (`pfs`, `avs`, `mds`, `eds`, `tvs`, `sms`) and
fallback prefixes generated by `signal_type.value[:3]` for signals
without explicit recommenders.

**Reason**: Signals like `doc_impl_drift`, `broken_encapsulation`,
`test_production_divergence` etc. produce tasks via the generic path.
Their IDs use a 3-char prefix derived from the signal value (e.g.,
`doc`, `bro`, `tes`).  Rejecting these would cause false quality
failures on real benchmark data.

## D7: Phase B Tolerates Zero Findings

**Decision**: If a real benchmark JSON contains only findings with signal
types or severities that cannot be mapped to the current enum values,
Phase B reports 0 tasks rather than failing the benchmark.

**Reason**: Benchmark JSONs may have been generated by older Drift
versions with different signal types.  Phase B's purpose is to validate
the *task generator*, not the *analysis engine*.  A zero-finding result
is logged as a quality score of `N/A` and does not count as a failure.

## D8: Atomic Repair Interventions

**Decision**: Each repair function modifies exactly one file (or a
minimal set of files) to address exactly one signal finding.

**Reason**: Atomic interventions isolate causality — we can attribute
score changes to the specific repair, not to side effects.  Sequential
application (repair MDS → measure → repair EDS → measure) creates a
clear chain of evidence.

## D9: Claim-Grenze — Translation + Verification, nicht Orchestrator

**Decision**: Der Benchmark wird als **Translation + Verification**
Nachweis gerahmt, nicht als vollständiger Repair-Orchestrator-Beweis.

**Reason**: Was nachweisbar ist:
- Drift erzeugt deterministische Repair-Tasks aus Analysebefunden
- Korrekte Reparaturen senken den Drift-Score messbar
- Falsche Reparaturen werden reproduzierbar abgelehnt

Was *nicht* nachgewiesen ist:
- Echte Coding-Agents führen Tasks autonom in Produktionsrepos aus
- Multi-Step-Reparaturen über abhängige Findings hinweg
- Vergleichsvorsprung gegenüber ungesteuerter Agentenarbeit

Diese Trennung ist strategisch gewollt.  Ein zu breiter Claim kostet
Glaubwürdigkeit (Policy §6: Glaubwürdigkeit > Features).  Ein präziser
enger Claim ist aktuell stärker als ein ungeprüfter breiter.

## D10: Explizite Verifikationsmetriken (FAR/FRR)

**Decision**: Der Summary enthält explizite FAR (False Acceptance Rate)
und FRR (False Rejection Rate) sowie TPR/TNR.

**Reason**: Nur `repair_success_rate` und `false_acceptance_rate` als
skalare Zusammenfassung reicht nicht.  Ein skeptischer Leser braucht:
- FAR: Wie oft akzeptiert Drift eine falsche Reparatur?
- FRR: Wie oft lehnt Drift eine korrekte Reparatur ab?
- TPR/TNR: Sensitivität und Spezifität der Verifikation

Diese vier Metriken machen die Verifikationsschärfe direkt prüfbar.

## D11: Determinismus-Check über wiederholte Runs

**Decision**: Jedes Repo wird 3× auf identischem State analysiert.
Scores, Finding-Counts und Task-Counts müssen identisch sein.

**Reason**: „Reproduzierbar" via Skript ≠ „deterministisch in der
Engine".  Der 3× Check beweist, dass die Analyse-Engine selbst bei
Wiederholung Bit-identische Ergebnisse liefert — keine versteckten
Zufallskomponenten in Scoring, Finding-Generierung oder
Task-Priorisierung.

## D12: Per-Signal-Deltas und Diff-Metriken

**Decision**: Jede Reparatur tracked (a) Score-Delta pro betroffenes
Signal, nicht nur den Gesamt-Score, und (b) Git-Diff-Statistiken
(Dateien, Insertions, Deletions).

**Reason**: Gesamt-Score-Delta zeigt nur „etwas wurde besser", nicht
*was* besser wurde.  Per-Signal-Deltas zeigen kausalen strukturellen
Effekt.  Diff-Größe zeigt operative Realisierbarkeit: Ein Agent der
3 Dateien und 28 Zeilen ändern muss, ist operationell anders als einer
der 50 Dateien braucht.

## D13: TVS-Side-Effects als bekannte Limitation

**Decision**: TVS-Side-Effects (Temporal Volatility durch Repair-Commits)
werden explizit als `known_limitation` in `claim_boundary` dokumentiert
und durch einen dedizierten Phase-C-Kaskadentest kontrolliert.

**Reason**: Jede MDS-Reparatur erzeugt einen Git-Commit.  Der Commit
verändert die Git-Historie und kann neue `temporal_volatility`-Findings
erzeugen.  Das ist kein Bug im Benchmark, sondern eine reale Eigenschaft
von Drift: temporale Signale reagieren auf Codeänderungen.  Ein Agent
der sequentiell repariert, muss damit rechnen.

Der Kaskadentest (Phase C) zeigt:
- TVS-Findings treten nach Repair-Commits auf (erwartetes Verhalten)
- Sie divergieren nicht unbegrenzt (kein Kaskaden-Risiko)
- Aber sie verzerren das Gesamt-Score-Delta einzelner Reparaturen

Nicht als Bug, sondern als dokumentierte Eigenschaft behandelt.

## D14: EDS-Verifikations-Semantik klargestellt

**Decision**: Das Feld `finding_resolved` wird zu
`targeted_finding_resolved` umbenannt.  Zusätzlich wird
`net_finding_delta` (Differenz der Finding-Counts vor/nach Repair)
erfasst.  Wenn das gezielte Finding verschwunden ist aber der
Signal-Finding-Count gleich bleibt, wird ein `side_effect_note`
hinzugefügt.

**Reason**: Das bisherige `finding_resolved: true` bei
`post_repair_signal_findings: 1` war für externe Prüfer verwirrend.
Es entstand, weil das EDS-Repair das gezielte Finding (transform_records
mit hoher Komplexität) tatsächlich beseitigt, aber ein neues, niedrigeres
EDS-Finding für verbleibende Komplexität (z.B. Parameteranzahl) entsteht.

Die neue Semantik macht den Unterschied explizit:
- `targeted_finding_resolved`: Das spezifische Finding ist weg
- `net_finding_delta`: Gesamtveränderung der Signal-Finding-Anzahl
- `side_effect_note`: Erklärung wenn beides divergiert

## D15: DIA- und PFS-Repairs für Signal-Coverage

**Decision**: Signal-Coverage von 2/6 auf 4/6 erweitert durch:
- DIA correct (webapp): README-Phantomreferenzen bereinigen
- DIA incorrect (webapp): Phantomreferenzen austauschen statt entfernen
- PFS correct (webapp): Error-Handling-Muster konsolidieren

**Reason**: 2/6 war die sichtbarste Lücke im Benchmark.  Beide Repos
enthielten bereits 3 DIA-Findings und 1 PFS-Finding die nie repariert
wurden.  Die Repairs nutzen bestehende Findings — kein neues Repo nötig.

DIA wurde gewählt weil:
- 3 Findings pro Repo (höchste Count neben dem reparierten MDS)
- Repair ist klar definierbar (Phantom-Dir-Refs entfernen/korrigieren)
- Incorrect-Case natürlich konstruierbar (alte Phantoms ersetzen, neue einführen)

PFS wurde gewählt weil:
- webapp hat explizit 4 Error-Handling-Varianten injiziert
- Konsolidierung ist ein realistischer Agenten-Task
- Zeigt Cross-File-Repair (4 Dateien gleichzeitig ändern)

## D16: FAR/FRR-Stichprobe vergrößert

**Decision**: Von n=3 correct / n=1 incorrect auf n=5 correct /
n=3 incorrect erweitert.  Stichprobengröße wird explizit in
`verification_metrics.sample_sizes` dokumentiert.

**Reason**: n=1 für den Negativ-Fall war statistisch nicht belastbar.
Ein skeptischer Reviewer sieht sofort, dass FAR=0% bei n=1
nicht verallgemeinerbar ist.

Die Erweiterung auf n=3 incorrect (MDS, DIA, EDS) verteilt die
Negativfälle über verschiedene Signaltypen, was die
Verifikationsbreite erhöht.  n=5 correct (MDS×2, EDS, DIA, PFS)
deckt 4 von 6 Recommender-Signalen ab.

Verbleibendes Defizit: n<10 pro Klasse reicht nicht für formale
statistische Signifikanz.  Das wird in `known_limitations` transparent
als Limitation dokumentiert, nicht als Schwäche versteckt.
